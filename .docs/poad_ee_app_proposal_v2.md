# ข้อเสนอโครงการพัฒนาแอปพลิเคชัน (Project Proposal)

## 1. ข้อมูลทั่วไปของโครงการ (Project Overview)
* **ชื่อโครงการ:** โครงการพัฒนาแอปพลิเคชัน "ปวดอึ" (Poad-Ee)
* **ประเภทผลิตภัณฑ์:** Progressive Web App (PWA) / Hybrid Mobile Application
* **กลุ่มเป้าหมายหลัก:** ผู้ที่ใส่ใจในสุขภาพระบบขับถ่าย, ผู้ที่มีปัญหาลำไส้แปรปรวน (IBS), ท้องผูกเรื้อรัง, คุณแม่ที่ต้องการติดตามการขับถ่ายของทารก, และบุคคลทั่วไปที่ต้องการ Tracker สุขภาพที่ใช้งานง่ายและเข้าถึงได้ทันที

---

## 2. หลักการและเหตุผล (Problem Statement & Justification)
ระบบขับถ่ายเป็นดัชนีชี้วัดสุขภาพที่สำคัญที่สุดอย่างหนึ่งของร่างกาย (Key Health Indicator) สัญญาณของโรคร้ายแรงหลายชนิด เช่น มะเร็งลำไส้ใหญ่, ภาวะลำไส้อักเสบ หรือภาวะขาดน้ำ สามารถตรวจพบได้ล่วงหน้าผ่านความถี่ ลักษณะ และสีของอุจจาระ 

อย่างไรก็ตาม แอปพลิเคชันในตลาดปัจจุบันมักใช้ภาษาที่เป็นทางการเกินไป มีขั้นตอนการบันทึกที่ซับซ้อน หรือใช้ชื่อภาษาอังกฤษที่เข้าใจยากสำหรับผู้ใช้งานทั่วไป แอปพลิเคชัน **“ปวดอึ”** จึงถูกออกแบบมาภายใต้แนวคิด **"Simple, Fast, and Functional"** เพื่อทลายกำแพงเหล่านั้นด้วยชื่อที่ตรงไปตรงมา เข้าถึงง่าย และระบบบันทึกที่เสร็จสิ้นได้ภายใน 3 คลิก

---

## 3. วัตถุประสงค์ (Project Objectives)
1. เพื่อพัฒนาแอปพลิเคชันที่มี UI/UX เป็นมิตร สามารถบันทึกข้อมูลการขับถ่ายได้อย่างรวดเร็ว (ภายในระยะเวลาไม่เกิน 10 วินาทีต่อครั้ง)
2. เพื่อประยุกต์ใช้ **Bristol Stool Chart** (เกณฑ์มาตรฐานทางการแพทย์) ให้กลายเป็นภาพกราฟิกที่เข้าใจง่ายสำหรับการประเมินสุขภาพลำไส้
3. เพื่อสร้างระบบสรุปผลและวิเคราะห์ข้อมูลเชิงลึก (Analytics Dashboard) ที่ผู้ใช้สามารถนำไปแสดงต่อแพทย์ผู้เชี่ยวชาญได้ทันทีเมื่อเกิดความผิดปกติ
4. เพื่อรองรับการเข้าถึงแบบไร้รอยต่อผ่านเทคโนโลยี PWA ทำให้ผู้ใช้สามารถเพิ่มแอปไปยังหน้าจอโฮม (**Add to Home Screen - A2HS**) ได้ทันทีโดยไม่ต้องผ่าน App Store

## 4. คุณสมบัติและฟังก์ชันหลัก (Core Features)

### 4.1 Quick Log (ระบบบันทึกด่วน)
* **Time & Date:** บันทึกเวลาโดยอัตโนมัติ (Default: Current Time Zone, เช่น GMT+7 สำหรับประเทศไทย)
* **Stool Type:** เลือกรูปทรงอุจจาระตามเกณฑ์ Bristol Stool Chart (Type 1 - Type 7) ผ่านการแสดงผลด้วยไอคอนที่น่ารักและเข้าใจง่าย
* **Color & Amount:** บันทึกสี (เช่น น้ำตาล, เขียว, ขอดำ, แดงปน) และปริมาณ (น้อย, ปกติ, มาก)
* **Effort Level:** บันทึกระดับความยากในการเบ่ง (เบ่งสบาย, ต้องออกแรง, ท้องผูก/เจ็บ)

### 4.2 Dashboard & Trends (ระบบวิเคราะห์ผล)
* **Health Score:** คำนวณคะแนนสุขภาพลำไส้รายสัปดาห์/รายเดือน
* **Frequency Tracker:** แสดงกราฟความถี่เพื่อตรวจหาภาวะท้องผูกหรือท้องเสียเรื้อรัง
* **Warning System:** ระบบแจ้งเตือนอัจฉริยะเมื่อพบความผิดปกติ เช่น ไม่ถ่ายติดต่อกันเกิน 3 วัน หรือมีสัญญาณสีที่สุ่มเสี่ยง (เช่น สีดำสนิท หรือมีเลือดปน)

### 4.3 Export & Import (การนำเข้า/ส่งออกข้อมูล)
* **Export Data:** ส่งออกข้อมูลการบันทึกทั้งหมดในรูปแบบ JSON เพื่อใช้สำรองข้อมูล หรือย้ายข้อมูลไปยังอุปกรณ์อื่น
* **Import Data:** นำเข้าข้อมูลจากไฟล์ JSON ที่ส่งออกไว้ โดยระบบจะตรวจสอบความซ้ำซ้อนของข้อมูลอัตโนมัติ (Deduplication) — หากข้อมูลเดิมมีอยู่แล้วจะข้ามไป, หากเป็นข้อมูลใหม่จะเพิ่มเข้าไปในระบบ
* **Medical Report PDF:** ส่งออกรายงานสรุปพฤติกรรมการขับถ่ายในรูปแบบ PDF เพื่อใช้ยื่นให้แพทย์เฉพาะทางประกอบการวินิจฉัยโรคระบบทางเดินอาหาร

## 5. สถาปัตยกรรมทางเทคนิคและการรองรับ iOS A2HS (Technical Architecture & iOS Customization)

เพื่อให้ระบบมีความลื่นไหล ปลอดภัย รองรับการใช้งานแบบ Local-first และติดตั้งใช้งานง่ายผ่านเว็บบราวเซอร์โดยเฉพาะบนระบบปฏิบัติการ iOS จึงออกแบบสถาปัตยกรรมดังนี้:

### 5.1 Tech Stack
* **Frontend:** React (Vite) ร่วมกับ Tailwind CSS สำหรับการพัฒนาเว็บแอปพลิเคชันประสิทธิภาพสูง
* **Deployment:** GitHub Pages (`gh-pages`) สำหรับ Deploy โดยตรงจาก Repository โดยไม่ต้องพึ่งพา Third-party Platform
* **Local Storage:** `localStorage` จัดเก็บข้อมูลดิบและประวัติลงในเครื่องผู้ใช้โดยตรง เพื่อให้สามารถเปิดใช้งานและบันทึกได้ทันทีแม้อยู่ในจุดอับสัญญาณ
* **Time Management:** Strict ISO 8601 (UTC/Local) จัดการ Timezone อย่างเคร่งครัด (+07:00) ป้องกันปัญหาข้อมูลข้ามวันหรือเวลาเพี้ยน

### 5.2 การปรับแต่งเพื่อรองรับ iOS Add to Home Screen (A2HS)
เนื่องจาก iOS (Safari) มีเงื่อนไขการทำงานของ PWA และ A2HS ที่เฉพาะเจาะจง โครงการนี้จึงจะดำเนินการติดตั้ง Web App Manifest และ Apple Custom Tags ดังต่อไปนี้:
* **Apple Touch Icons:** จัดเตรียมไอคอนแอปพลิเคชันขนาดต่าง ๆ (เช่น 180x180, 192x192) และประกาศผ่าน `<link rel="apple-touch-icon" href="...">` เพื่อให้แสดงผลไอคอนบนหน้าจอโฮมของ iOS อย่างคมชัด ไม่เบลอ
* **PWA Web App Manifest:** ตั้งค่าฟิลด์ `display: "standalone"` เพื่อให้เมื่อเปิดแอปจากหน้าจอโฮม จะแสดงผลแบบเต็มหน้าจอ (Full Screen) ซ่อนแถบ URL และปุ่มควบคุมของ Safari ให้ความรู้สึกเหมือนเป็น Native App
* **iOS Status Bar Customization:** ตั้งค่าแท็ก `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">` เพื่อให้แถบสถานะด้านบนกลมกลืนไปกับธีมของแอปพลิเคชัน
* **Custom iOS A2HS Prompt:** เนื่องจาก iOS ไม่รองรับ Web API `beforeinstallprompt` (ไม่สามารถคลิกปุ่มติดตั้งบนหน้าเว็บแล้วเด้งหน้าต่างติดตั้งอัตโนมัติแบบ Android ได้) ระบบจะพัฒนา **Custom Overlay Prompt UI** ขึ้นมาโดยเฉพาะ เพื่อตรวจจับว่าผู้ใช้เข้าใช้งานผ่าน iOS Safari หรือไม่ หากใช่ จะแสดงคำแนะนำพร้อมแอนิเมชันชี้ไปที่ปุ่ม **"Share" (แชร์)** แล้วเลือก **"Add to Home Screen" (เพิ่มไปยังหน้าจอโฮม)** เพื่อให้กระบวนการติดตั้ง (Onboarding) ของผู้ใช้ iOS เป็นไปอย่างราบรื่นที่สุด

## 6. แผนการดำเนินงาน (Project Timeline)
ระยะเวลาดำเนินงานรวมประมาณ 12 สัปดาห์ (3 เดือน) แบ่งเป็นเฟส ดังนี้:

* **Week 1-2 [Design Phase]:** ออกแบบ UI/UX, โครงสร้างฐานข้อมูล และออกแบบภาพคำแนะนำการติดตั้ง A2HS สำหรับ iOS/Android
* **Week 3-8 [Development Phase]:** พัฒนาฟังก์ชันหน้า Log, ระบบคำนวณตามหลัก Bristol Stool Chart, การติดตั้ง Service Worker สำหรับ Offline Mode และปรับแต่ง Manifest/Meta tags ของ iOS
* **Week 9-10 [Testing & QA]:** ทำการทดสอบระบบบนอุปกรณ์ iOS จริง (Physical Devices) ทั้งบน Safari และเว็บบราวเซอร์อื่น ๆ เพื่อตรวจสอบการแสดงผลหน้าจอแบบ Standalone และการแจ้งเตือน
* **Week 11-12 [Deployment]:** Deploy ระบบขึ้น GitHub Pages ผ่าน `gh-pages` พร้อมเปิดให้ผู้ใช้งานสามารถติดตั้งผ่านการ Add to Home Screen ได้ทันที

## 7. ประโยชน์ที่คาดว่าจะได้รับ (Expected Benefits)
1. **ผู้ใช้งานทั่วไป:** สามารถเข้าถึงแอปพลิเคชันเพื่อบันทึกข้อมูลได้อย่างรวดเร็วทันใจผ่านหน้าจอโฮม โดยไม่เปลืองพื้นที่จัดเก็บข้อมูลในเครื่องแบบ Native App
2. **ความคล่องตัวในการพัฒนา:** ลดระยะเวลาและต้นทุนในการ Deploy เนื่องจากไม่ต้องรอการอนุมัติ (App Review) จาก Apple App Store ทำให้สามารถปรับปรุงฟีเจอร์และแก้ไขบั๊กได้อย่างรวดเร็ว
3. **ผู้ป่วยกลุ่มโรคระบบทางเดินอาหาร:** มี Data Log ที่เที่ยงตรง แม่นยำ ไม่ต้องใช้ความจำในการตอบคำถามแพทย์ ซึ่งช่วยให้กระบวนการรักษาตรงจุดและรวดเร็วยิ่งขึ้น
